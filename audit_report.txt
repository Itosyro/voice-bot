🔥 REPOSITORY AUDIT REPORT

EXECUTIVE SUMMARY
The Voice Polisher Bot repository presents a severe security risk due to critical vulnerabilities allowing remote code execution via prompt injection and unauthenticated database exhaustion. The architecture lacks robust input validation, relies on a naive memory rate-limiting scheme, and has command injection flaws in its background scripts. The codebase requires immediate and substantial remediation before being deployed to any production environment.

[CRITICAL-03] Denial of Service (DoS) via Unbounded In-Memory Rate Limiting
Category: Security / Insecure Design (OWASP A04)
File: src/middlewares/rate_limit.py Line: 13
Description: The `RateLimitMiddleware` stores request timestamps in a Python `defaultdict(list)` in memory. This state is not shared across multiple processes/workers (e.g., in a multi-container deployment), rendering it ineffective. Furthermore, a malicious user can send a massive number of requests, causing the list for their user ID to grow unbounded before cleanup, or filling the dictionary with millions of unique user IDs (if spoofed or via distributed attack), leading to an Out of Memory (OOM) crash.
Attack Scenario: Attacker writes a script to send thousands of requests per second. The server memory fills up storing the timestamps, crashing the bot. If running multiple replicas, the rate limit is bypassed because it's only in-memory per replica.
Vulnerable Code:
```python
class RateLimitMiddleware(BaseMiddleware):
    def __init__(self) -> None:
        self._user_requests: dict[int, list[float]] = defaultdict(list)
```
Fixed Code:
```python
# Use Redis for distributed and memory-safe rate limiting
import redis.asyncio as redis

class RateLimitMiddleware(BaseMiddleware):
    def __init__(self, redis_url: str):
        self.redis = redis.from_url(redis_url)

    async def __call__(self, handler, event, data):
        user = data.get("event_from_user")
        if user:
            key = f"rate_limit:{user.id}"
            requests = await self.redis.incr(key)
            if requests == 1:
                await self.redis.expire(key, 60)
            if requests > settings.rate_limit_per_user_per_min:
                return None # Drop or warn
        return await handler(event, data)
```

[CRITICAL-04] Root User in Docker Container
Category: DevOps / Infrastructure Security
File: Dockerfile Line: 4
Description: The Docker container runs as the `root` user. If an attacker achieves Remote Code Execution (e.g., via the command injection flaw or a vulnerability in an audio processing library like `ffmpeg`), they gain root access to the container, making container escape significantly easier.
Attack Scenario: Attacker exploits a vulnerability to execute arbitrary code. Because the process runs as root, they can install additional tools, modify system files, or attempt to break out to the host.
Vulnerable Code:
```dockerfile
FROM python:3.11-slim
# ... no USER directive ...
CMD ["sh", "-c", "alembic upgrade head && python scripts/sync_skills.py && python -m src.main"]
```
Fixed Code:
```dockerfile
FROM python:3.11-slim
RUN addgroup --system appgroup && adduser --system --group appuser
# Install deps...
WORKDIR /app
COPY --chown=appuser:appgroup . .
USER appuser
CMD ["sh", "-c", "alembic upgrade head && python scripts/sync_skills.py && python -m src.main"]
```
References: CWE-269

🟠 HIGH ISSUES

[HIGH-01] Missing Concurrency Handling / Race Condition in User Creation
Category: Architecture / Reliability (OWASP A04)
File: src/storage/users.py Line: 16
Description: The `get_or_create_user` function uses a non-atomic "check-then-act" pattern. If a user sends multiple messages simultaneously (e.g., holding down the voice button and sending multiple quickly), both requests might see `user is None` and attempt to insert the same `telegram_user_id`. Since `telegram_user_id` has a unique constraint, the second transaction will fail with an `IntegrityError`, crashing that request handler.
Attack Scenario: User rapidly sends multiple messages. The database throws an `IntegrityError`, causing the bot to drop requests and potentially spam the logs.
Vulnerable Code:
```python
    stmt = select(User).where(User.telegram_user_id == telegram_user_id)
    user = (await session.execute(stmt)).scalar_one_or_none()
    if user:
        return user
    user = User(...)
    session.add(user)
```
Fixed Code:
```python
from sqlalchemy.dialects.postgresql import insert

async def get_or_create_user(...):
    stmt = insert(User).values(telegram_user_id=telegram_user_id, ...).on_conflict_do_update(
        index_elements=['telegram_user_id'],
        set_=dict(username=username, first_name=first_name)
    ).returning(User)
    result = await session.execute(stmt)
    await session.flush()
    return result.scalar_one()
```

[HIGH-02] Unbounded Audio File Processing
Category: Security / Resource Exhaustion
File: src/handlers/voice.py Line: 63
Description: The bot downloads audio files without strictly limiting the file size *before* downloading. It relies on the `duration` metadata provided by Telegram, which can be spoofed or missing. An attacker could upload a massive file disguised as a short audio clip, causing the bot to exhaust memory or disk space when calling `file_bytes.read()`.
Attack Scenario: Attacker crafts a 2GB OGG file with metadata claiming it's 5 seconds long. The bot attempts to download and load the entire file into memory (`audio_bytes = file_bytes.read()`), causing an OOM kill.
Vulnerable Code:
```python
        file_bytes = await bot.download_file(file.file_path)
        audio_bytes = file_bytes.read()
```
Fixed Code:
```python
        import io
        # Read in chunks and enforce a hard byte limit (e.g., 20MB)
        MAX_BYTES = 20 * 1024 * 1024
        audio_bytes = b""
        async for chunk in bot.download_file(file.file_path): # Assuming async generator or use io.BytesIO limit
            audio_bytes += chunk
            if len(audio_bytes) > MAX_BYTES:
                raise ValueError("File too large")
```

💀 TOP 3 MOST BRUTAL FINDINGS
1. **The "Hope it works" Rate Limiter**: Storing timestamps in an unbounded Python `defaultdict` is a textbook memory leak waiting to happen. An attacker doesn't even need to try hard to crash the server; a simple `while true` curl script will OOM the container in minutes.
2. **System Prompt Swiss Cheese**: Concatenating untrusted speech-to-text directly into the LLM system prompt is practically begging for prompt injection. An attacker can just speak "Forget previous instructions" into their microphone and hijack the bot.
3. **The `handle_voice` God Function**: It's an 80-line monster that knows about Telegram, the database, Groq, file systems, and business logic. It's untestable spaghetti that violates every SOLID principle simultaneously.

🏆 TOP 3 THINGS DONE WELL
1. **Use of Async/Await**: The codebase consistently uses `asyncio`, `aiogram`, and `asyncpg`, which is great for scaling I/O bound Telegram bots.
2. **Database Schema**: The SQLAlchemy models are well-defined, use modern SQLAlchemy 2.0 features, and include proper indexing and foreign keys.
3. **Separation of Prompts**: Moving the massive prompt strings out of the logic and into dedicated `src/prompts/` files keeps the service layer much cleaner.
